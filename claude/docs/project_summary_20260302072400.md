# llmos-v0 — Project Summary

## What is this?

llmos-v0 is an "OS-level substrate for LLM agents." It provides the core primitives that LLM-powered agents need to run: task management, a shared store, an AI interface, and a toolbox. Think of it as a microkernel for AI agents — minimal, composable, and observable.

The goal is not to be a framework that tells you how to build agents, but to provide the low-level building blocks (like an OS provides processes, memory, and syscalls) that any agent architecture can build on.

## Core Primitives

### Store (`src/store.ts`)
An in-memory key-value store with hierarchical path-based keys (like a filesystem). Keys look like `/kernel/tasks/{id}/meta` or `/global/chunks/0`.

The key abstraction is **scoped accessors** — you create a view of the store that auto-prefixes all keys. A scope at `/kernel/` means `.set('tasks/abc/meta', data)` actually writes to `/kernel/tasks/abc/meta`. This gives different parts of the system isolated views without separate storage instances.

API: `get`, `set`, `delete`, `list` (by prefix), `append` (to arrays), `snapshot` (dump to JSON), `load` (restore from JSON).

### Tasks (`src/task.ts`)
A task is a named async function with a typed event (input) and typed result (output). Tasks run concurrently and can spawn child tasks. Each task gets a `TaskContext` with:
- **Scoped stores**: `local` (private to this task), `ephemeral` (auto-cleaned on completion), `global` (shared), `raw` (full access)
- **AI interface**: scoped to the task for usage tracking
- **Toolbox**: registered tool functions
- **Spawn**: ability to create child tasks

The kernel automatically tracks task metadata (status, timing, errors) and cleans up ephemeral data.

### AI Interface (`src/ai.ts`)
Wraps LLM API calls with automatic logging and per-task usage tracking. Every request/response is recorded in the kernel store at `/kernel/ai/requests`. Token usage is accumulated per task at `/kernel/tasks/{id}/usage`.

### Toolbox (`src/toolbox.ts`)
A registry of named tool functions with JSON schemas. Tools can be registered, listed, and executed. All invocations are logged to the kernel store. Tools can be passed to the AI interface for function-calling.

### System Entry Point (`src/index.ts`)
`createLLMOS(config)` wires everything together — creates a Store, sets up the kernel scope, initializes AI and Toolbox, and returns a spawn function. This is the main entry point for using the system.

## Key Conventions

Store paths follow a hierarchical namespace convention:
```
/kernel/tasks/{taskId}/meta      — task metadata (managed by the system)
/kernel/tasks/{taskId}/result    — task return values
/kernel/tasks/{taskId}/usage     — per-task AI token usage
/kernel/ai/requests              — all AI request/response logs
/kernel/toolbox/invocations      — tool invocation logs
/global/...                      — shared application data
/task/{taskId}/...               — task-local storage
/ephemeral/tasks/{taskId}/...    — auto-cleaned on task completion
```

The `/kernel/` namespace is system-managed. Everything else is application space.

## Demo: Book Search Agent (`examples/book-search/`)

A working demo that searches through "The Hard Thing About Hard Things" using a map-reduce pattern. This demonstrates all the core primitives in action.

### How it works
1. **Extract**: Parse epub into plain text
2. **Chunk**: Split into ~50K character chunks (paragraph-boundary splitting)
3. **Coordinator task**: Receives a search query, reads chunk metadata from the global store, spawns one worker per chunk
4. **Worker tasks** (10 in parallel): Each reads its chunk from the global store, asks the AI to evaluate relevance, returns relevant passages
5. **Synthesis**: Coordinator collects results, asks the AI to synthesize a coherent answer

### Mock AI
For testing without API costs, `MockAI` does keyword matching — if the query words appear in the chunk, it's "relevant." This is structurally identical to real AI usage (same interface, same logging, same token tracking) but deterministic and free. A configurable delay (`MOCK_DELAY_MS`) simulates real latency for UI testing.

### Server (`server/src/server.ts`)
Hono web server with:
- `POST /api/search { query }` — runs a search, returns results
- `GET /api/state` — returns full store snapshot (polled by UI)

### React UI (`ui/`)
Two-column layout built with Tailwind v4 + shadcn/ui, matching the armandmcqueen.dev design (light theme, teal accents, Geist fonts).

**Left column**: Search input + live progress (ActiveProcesses component shows workers completing in real-time with progress bar) + results (answer + relevant passages)

**Right column**: Debug panels via tabbed interface:
- **Tasks**: Table of all tasks with status, parent, duration
- **Store**: Interactive tree explorer of the full store state
- **AI Log**: Expandable cards showing every AI request/response
- **Usage**: Token usage bar chart per task

The UI polls `/api/state` every 500ms during search and renders from the store snapshot. The `useStoreState` hook plus helper functions (`extractTasks`, `extractAIRequests`, `extractTaskUsage`) parse the flat snapshot into typed data structures.

### CLI (`server/src/cli.ts`)
Commander-based CLI with `search`, `extract`, and `inspect` subcommands. The `inspect` command reads `store-snapshot.json` and provides summary/ls/tree/get/dump views of the store state.

## Tech Stack

- **Core**: TypeScript, ESM modules, vitest for testing
- **AI**: Anthropic SDK via Vercel AI SDK (`ai` + `@ai-sdk/anthropic`)
- **Server**: Hono + @hono/node-server
- **UI**: React 19, Vite, Tailwind CSS v4, shadcn/ui, Geist fonts (via `non.geist`)
- **Monorepo**: pnpm workspaces

## Current State (2026-03-02)

Everything is on the `first_demo` branch. The core primitives work, the demo runs end-to-end, and the UI visualizes the full execution. No tests exist yet (vitest is configured but no test files).

### Known Issues / Next Steps

- **Execution isolation**: Running a second search shows stale data from the first. A store namespacing proposal exists at `claude/designs/proposal_store_namespacing.md` — the idea is UUID-identified namespaces that transparently prefix all store paths, giving each execution a clean isolated view. The exact design of storage layers is still being considered.
- **No tests**: The testing infrastructure (vitest) is configured but no tests have been written yet.
- **Old CLIs**: Some standalone CLI entry points in epub.ts, agent.ts, store-cli.ts still have raw `process.argv` parsing alongside the unified cli.ts.
